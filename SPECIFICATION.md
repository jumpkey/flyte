# Flyte — Project Specification

## Overview

**Flyte** is an architectural proof-of-concept for a [fly.io](https://fly.io)-hosted web application. It demonstrates a clean separation between a web application layer and a data services layer, both written in TypeScript, with HTMX driving the UI and PostgreSQL as the backing store. The application implements a complete user lifecycle: registration, email verification, login, password reset via email link, session management, profile editing, and logout, along with event logging for user actions.

This spec is the single source of truth for autonomous code generation. Follow it literally.

## Architecture

```
┌─────────────────────────────────────────────────────┐
│  Browser (HTMX + minimal JS)                        │
└──────────────┬──────────────────────────────────────┘
               │  HTML fragments over HTTP
┌──────────────▼──────────────────────────────────────┐
│  Web App Layer (TypeScript / Hono or Express)        │
│  - Route handlers                                    │
│  - Server-side HTML rendering (template engine)      │
│  - Session management                                │
│  - CSRF protection                                   │
│  - Input validation                                  │
└──────────────┬──────────────────────────────────────┘
               │  Internal function calls
┌──────────────▼──────────────────────────────────────┐
│  Data Services Layer (TypeScript)                    │
│  - User CRUD                                         │
│  - Auth logic (hashing, token generation)            │
│  - Email dispatch                                    │
│  - Event logging                                     │
└──────────────┬──────────────────────────────────────┘
               │  SQL via pg / postgres.js
┌──────────────▼──────────────────────────────────────┐
│  PostgreSQL (fly.io managed or Supabase)             │
└─────────────────────────────────────────────────────┘
```

### Key Constraints

- **No client-side framework.** All interactivity is driven by HTMX attributes on server-rendered HTML. The only JS in the browser is the HTMX library and optional small inline scripts for progressive enhancement (e.g., password strength meter).
- **Two logical layers, one process.** The web app layer and data services layer live in the same Node.js process. They are separated by module boundaries (distinct directories, explicit imports), not by a network boundary. This keeps deployment trivial while enforcing architectural discipline.
- **TypeScript everywhere.** Both layers use strict TypeScript (`"strict": true` in tsconfig).
- **Fly.io deployment target.** The app runs in a Docker container on fly.io. The Dockerfile and `fly.toml` are part of the deliverable.

## Database Schema

All timestamps are `TIMESTAMPTZ`. All tables use `UUID` primary keys generated by `gen_random_uuid()`.

| Column | Type | Constraints | Notes |
|---|---|---|---|

| `password_hash` | `VARCHAR(255)` | NOT NULL | bcrypt hash |
| `display_name` | `VARCHAR(100)` | NOT NULL | |
| `is_verified` | `BOOLEAN` | NOT NULL, default `FALSE` | Set to TRUE when email verification link is clicked |
| `is_admin` | `BOOLEAN` | NOT NULL, default `FALSE` | |

### Migration File Naming

Use sequential numbered files: `001_create_users.sql`, `002_create_login_events.sql`, etc. Each file contains both `-- migrate:up` and `-- migrate:down` sections.

### Trigger

Create an `updated_at` trigger function applied to the `users` table:

```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at();
```

## Application Routes

### Public Routes (no auth required)

| Method | Path | Handler | Description |
|---|---|---|---|
| `GET` | `/` | `homeController.index` | Public home page. If the user has a valid session, redirect to `/dashboard`. |
| `GET` | `/login` | `authController.loginForm` | Render login form. |
| `POST` | `/login` | `authController.login` | Validate credentials, create session, redirect to `/dashboard`. |
| `GET` | `/register` | `authController.registerForm` | Render registration form. |
| `POST` | `/register` | `authController.register` | Create user, send verification email, render "check your email" page. |
| `GET` | `/verify-email?token=<token>` | `authController.verifyEmail` | Validate token, mark user verified, auto-login, redirect to `/dashboard`. |
| `POST` | `/api/check-email` | `authController.checkEmail` | **HTMX endpoint.** Returns an HTML fragment indicating whether the email is already registered. Used for inline validation on the registration form. |
| `GET` | `/forgot-password` | `authController.forgotPasswordForm` | Render forgot-password form (email field only). |
| `POST` | `/forgot-password` | `authController.forgotPassword` | Generate reset token, send reset email, render confirmation page. |
| `GET` | `/reset-password?token=<token>` | `authController.resetPasswordForm` | Validate token, render new-password form. |
| `POST` | `/reset-password` | `authController.resetPassword` | Validate token + new password, update password hash, redirect to `/login`. |

### Protected Routes (auth required — redirect to `/login` if no session)

| Method | Path | Handler | Description |
|---|---|---|---|
| `GET` | `/dashboard` | `dashboardController.index` | Logged-in home page. Shows user greeting and nav. |
| `GET` | `/profile` | `profileController.editForm` | Render profile edit form (display name, password change). Email is shown but **not editable**. |
| `POST` | `/profile` | `profileController.update` | Validate and update profile fields. Return HTMX fragment with success/error feedback. |
| `POST` | `/logout` | `authController.logout` | Destroy session, redirect to `/`. |

### Middleware Stack

1. **Request logger** — log every request (method, path, status, duration) via pino.
2. **Session middleware** — parse/validate session cookie.
3. **CSRF middleware** — inject token into forms; validate on POST.
4. **Action logger** — for authenticated requests, write to `user_action_events`.
5. **Auth guard** — applied to protected routes; checks session validity.

## Page Specifications

### P1: Public Home Page (`/`)

- Minimal landing page with the app name, a one-line description, and links to **Log In** and **Register**.
- If the user is already logged in (valid session), redirect to `/dashboard`.

### P2: Login Page (`/login`)

- Form fields: **Email**, **Password**.
- A "Forgot your password?" link below the password field, linking to `/forgot-password`.
- A "Don't have an account? Register" link below the form.
- On failed login, re-render the form with an error message (generic: "Invalid email or password" — do not reveal which field was wrong).
- On success, redirect to `/dashboard`.
- Log every attempt to `login_events`.

### P3: Registration Page (`/register`)

- Form fields: **Email**, **Display Name**, **Password**, **Confirm Password**.
- The **Email** field uses HTMX to call `POST /api/check-email` on `blur` or after a short debounce, displaying inline feedback ("Email is available" / "This email is already registered").
  - HTMX attributes: `hx-post="/api/check-email"`, `hx-trigger="blur changed"`, `hx-target="#email-feedback"`, `hx-swap="innerHTML"`.
- Client-side: basic password match check (inline `<script>` or HTMX).
- Server-side validation on submit:
  - Email format valid.
  - Email not already taken (race condition: handle unique constraint violation gracefully).
  - Password meets minimum length (≥ 8 characters).
  - Passwords match.
- On validation failure, re-render form with field-specific error messages and previously entered values (except password).

### P4: Registration Handler (`POST /register`)

- Hash password with bcrypt (cost ≥ 12).
- Generate a random verification token (64 hex characters via `crypto.randomBytes(32).toString('hex')`).
- Store a **hashed** version of the token in the database (`verification_token` column).
- Set `verification_token_expires_at` to `NOW() + INTERVAL '24 hours'`.
- Insert user row. On unique constraint violation, return to form with error.
- Send verification email containing a link: `https://<APP_DOMAIN>/verify-email?token=<raw_token>`.
- Render a "Check your email" confirmation page (do **not** auto-login yet).
- Log a `user_action_event` with action `'registration'`.

### P5: Email Verification Handler (`GET /verify-email`)

- Extract `token` query parameter.
- Hash the token and look up the matching user where `is_verified = FALSE` and `verification_token_expires_at > NOW()`.
- If no match: render an error page ("Invalid or expired verification link. Please register again.").
- If match: set `is_verified = TRUE`, clear `verification_token` and `verification_token_expires_at`, create a session, redirect to `/dashboard`.
- Log a `user_action_event` with action `'email_verified'`.

### P6: Dashboard (`/dashboard`)

- Greeting: "Welcome, {display_name}".
- Navigation links: **Profile**, **Logout**.
- Placeholder content area for future feature modules.
- The logout link should be a `<form>` with `method="POST"` action to `/logout` (not a GET link) to avoid CSRF issues and accidental logouts via prefetch.

### P7: Profile Edit Page (`/profile`)

- Display the user's **email** as read-only text (not an input).
- Editable fields: **Display Name**.
- Password change section (all three required to change password):
  - **Current Password**
  - **New Password**
  - **Confirm New Password**
- Submit via HTMX `hx-post="/profile"` with `hx-target="#profile-feedback"` to show success/error inline without a full page reload.
- Server-side validation:
  - If changing password: verify current password, enforce minimum length on new password, confirm match.
  - Display name must not be empty.
- On success: flash "Profile updated" message.
- Log a `user_action_event` with action `'profile_update'`.

### P8: Forgot Password Page (`/forgot-password`)

- Form field: **Email**.
- On submit, **always** render the same confirmation page ("If an account with that email exists, we've sent a password reset link") regardless of whether the email is found. This prevents account enumeration.
- A "Back to login" link.

### P9: Forgot Password Handler (`POST /forgot-password`)

- Look up user by email.
- If user exists, is verified, and is not locked:
  - Generate a random reset token (64 hex characters via `crypto.randomBytes(32).toString('hex')`).
  - Store a **hashed** version (SHA-256) of the token in `password_reset_token`.
  - Set `password_reset_token_expires_at` to `NOW() + INTERVAL '1 hour'` (shorter than verification — password resets are time-sensitive).
  - Send reset email containing a link: `https://<APP_DOMAIN>/reset-password?token=<raw_token>`.
- If user does not exist, is unverified, or is locked: do nothing (no email sent), but render the same confirmation page. No timing difference — perform a dummy delay or constant-time comparison to avoid timing side channels.
- Log a `user_action_event` with action `'password_reset_requested'` (only if user exists).
- Rate limit: same IP-based limit as `/login` and `/register`.

### P10: Reset Password Page (`GET /reset-password?token=<token>`)

- Extract `token` query parameter.
- Hash the token and look up the matching user where `password_reset_token_expires_at > NOW()`.
- If no match: render an error page ("Invalid or expired reset link. Please request a new one." with a link to `/forgot-password`).
- If match: render a form with fields **New Password** and **Confirm New Password**, plus a hidden field carrying the token.

### P11: Reset Password Handler (`POST /reset-password`)

- Extract `token`, `password`, and `confirmPassword` from the request body.
- Hash the token, look up the matching user, validate expiry (same as P10).
- Server-side validation:
  - Token is valid and not expired.
  - Password meets minimum length (≥ 8 characters).
  - Passwords match.
- If valid:
  - Hash the new password with bcrypt.
  - Update `password_hash`, clear `password_reset_token` and `password_reset_token_expires_at`.
  - Reset `failed_login_count` to 0 and set `is_locked` to FALSE (a successful reset unlocks the account).
  - Invalidate all existing sessions for this user (if using DB-backed sessions, delete rows matching this user).
  - Redirect to `/login` with a flash message: "Password reset successfully. Please log in."
- If invalid: re-render the form with error messages.
- Log a `user_action_event` with action `'password_reset_completed'`.

## Security Requirements

1. **Passwords** are hashed with bcrypt, cost ≥ 12. Raw passwords are never stored or logged.
2. **Verification and password reset tokens** are stored hashed (SHA-256). The raw token appears only in the email URL.
3. **Sessions** use `httpOnly`, `secure` (in production), `sameSite: 'lax'` cookies.
4. **CSRF** tokens are validated on every state-changing request (POST, PUT, DELETE).
5. **Rate limiting** on `POST /login`, `POST /register`, and `POST /forgot-password` (e.g., 10 requests per minute per IP). Use a simple in-memory store for the PoC; note that this should be Redis-backed in production.
6. **Input sanitization**: all user input is parameterized in SQL queries (no string concatenation). HTML output is escaped by the template engine.
7. **Error messages** on login and forgot-password never reveal whether the email exists. Registration email-check is intentionally more revealing (UX tradeoff, common practice).
8. **Account lockout** after N consecutive failed logins. Locked accounts get a generic login error, not a "locked" message. A successful password reset unlocks the account.
9. **Helmet** (or equivalent) sets security headers: `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, etc.
10. **Password reset tokens expire after 1 hour.** Shorter than verification tokens because resets are more security-sensitive. A successful reset invalidates the token and all existing sessions for that user.

```html
<input type="email" name="email" id="email"
       hx-post="/api/check-email"
       hx-trigger="blur changed delay:500ms"
       hx-target="#email-feedback"
```

- `<span class="success">Email is available</span>`
- `<span class="error">This email is already registered</span>`

```html
<form hx-post="/profile" hx-target="#profile-feedback" hx-swap="innerHTML">
    <!-- fields -->
    <button type="submit">Save Changes</button>

<div id="profile-feedback"></div>
```

```html
<form method="POST" action="/logout" style="display:inline;">
```

## Seed Script (`scripts/seed.ts`)

The seed script creates a default admin user for immediate development use. It **must** be idempotent — safe to run repeatedly without creating duplicates or errors.

### Behavior

- Look up a user by `SEED_ADMIN_EMAIL` (default: `admin@flyte.local`).
- If the user does not exist, create one with:
  - `email`: value of `SEED_ADMIN_EMAIL`
  - `password`: hashed value of `SEED_ADMIN_PASSWORD` (default: `changeme123`)
  - `display_name`: `Admin`
  - `is_verified`: `TRUE` (skip email verification)
  - `is_admin`: `TRUE`
  - `is_locked`: `FALSE`
- If the user already exists, update the `password_hash` to match `SEED_ADMIN_PASSWORD` (in case the env var changed) and ensure `is_verified` and `is_admin` are `TRUE`. Use an `INSERT ... ON CONFLICT (LOWER(email)) DO UPDATE` (upsert).
- Log the result to stdout: `"Seeded admin user: admin@flyte.local"` or `"Admin user already exists, password updated."`.

### npm script

```json
{
  "scripts": {
    "seed": "tsx scripts/seed.ts"
  }
}
```

The seed script reads `DATABASE_URL` from the environment (or `.env` via dotenv). It is part of the standard dev setup: `npm run migrate && npm run seed`.

## GitHub Copilot Cloud Agent Environment

Copilot cloud agent runs in an ephemeral environment powered by GitHub Actions. It does **not** have Docker Compose. Instead, it provisions service containers via the `services:` key in the `copilot-setup-steps.yml` workflow. This means the agent **can** spin up a real PostgreSQL instance to run migrations, seed data, and execute the application during development.

### `.github/workflows/copilot-setup-steps.yml`

```yaml
name: "Copilot Setup Steps"

on:
  workflow_dispatch:
  push:
    paths:
      - .github/workflows/copilot-setup-steps.yml
  pull_request:
    paths:
      - .github/workflows/copilot-setup-steps.yml

jobs:
  copilot-setup-steps:
    runs-on: ubuntu-latest
    permissions:
      contents: read

    services:
      postgres:
        image: postgres:16-alpine
        ports: ["5432:5432"]
        env:
          POSTGRES_USER: flyte
          POSTGRES_PASSWORD: flyte
          POSTGRES_DB: flyte
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    env:
      DATABASE_URL: postgres://flyte:flyte@localhost:5432/flyte

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Run migrations
        run: npm run migrate

      - name: Seed admin user
        run: npm run seed
```

### Important Notes for the Agent

- **Do not use `container:` at the job level** in `copilot-setup-steps.yml`. There is a known incompatibility with Copilot's helper scripts when running inside a job-level container. Use `services:` only (as shown above), which runs the Postgres container alongside the default runner.
- **Do not use Docker Compose** in the setup steps. The agent environment does not have a Docker daemon available for `docker compose`. The `services:` key is the Actions-native equivalent.
- The `docker-compose.yml` in the project root is for **local human development only**, not for the agent.

## Agent Build Instructions

When using this spec to generate the project:

1. **Start with the database.** Write migration files first. Run them to verify the schema.
2. **Build the data services layer** (`src/services/`). These are pure functions over the database and have no web dependencies. They can be tested in isolation.
3. **Build the web layer bottom-up:** middleware first, then controllers, then views.
4. **Wire up the entry point** (`src/index.ts`) last.
5. **Write the seed script** (`scripts/seed.ts`). Run `npm run migrate && npm run seed` to verify.
6. **Verify each page** works before moving to the next. The order is: home → register → email check API → verify email → login → forgot password → reset password → dashboard → profile → logout.
7. **Do not skip CSRF or session middleware.** They are required for the app to function correctly, not optional hardening.
8. **Commit working code at each milestone** (schema done, services done, auth flow done, password reset done, profile done).
